Question 1 Amazon Bedrock Fundamentals
A team runs a nightly job that scores 4 million archived support transcripts with an Anthropic model on Bedrock. Latency is irrelevant, but the finance team demands the lowest possible per-token cost and the workload must not consume the on-demand rate limits shared by the interactive product. Which approach best fits?
  1. AUse Bedrock batch inference by uploading the records to Amazon S3 and creating a model invocation job
  2. BCreate a cross-region inference profile and loop InvokeModel over the records
  3. CSubmit the records through the on-demand Converse API in parallel across many threads
  4. DPurchase Provisioned Throughput with no commitment for the duration of the job
Show answer & explanation
Correct answer: A
Batch inference reads inputs from S3, runs asynchronously with no latency guarantee, is billed at a discount versus on-demand, and uses a separate quota so it does not compete with interactive traffic. On-demand parallelism consumes the shared rate limits; Provisioned Throughput is priced for sustained high-volume low-latency serving, not one-off cheap bulk jobs; cross-region profiles only distribute on-demand capacity.
Source: Amazon Bedrock — Batch inference

  4. DA custom proxy that intercepts every call and writes to a text file on each host
Show answer & explanation
Correct answer: B
Model invocation logging natively captures request and response data and delivers it to S3 and/or CloudWatch Logs for querying, with no custom code. A per-host proxy is heavy and error-prone, Guardrails are not a general logging source, and throughput metrics do not contain prompt text.

Question 75 Security, Cost & Operations
A team wants defense against a single compromised application credential invoking expensive models at scale, limiting blast radius without blocking legitimate low-cost usage. Which control is the most appropriate first line?
  1. AShare one powerful IAM role across all apps for simplicity
  2. BDisable Guardrails to reduce per-request overhead
  3. CApply least-privilege IAM scoped to only the needed models/actions, and use service quotas/throttling to cap request rates
  4. DGrant full bedrock access but enable invocation logging to detect abuse after the fact
Show answer & explanation
Correct answer: C
Least-privilege IAM restricting which models/actions a credential can use, combined with quotas/throttling to cap rates, directly limits blast radius while allowing legitimate use. Logging is detective (after the fact), shared powerful roles widen blast radius, and disabling Guardrails is unrelated to credential abuse.
