A company is preparing to invoke a newly released foundation model in Bedrock for the first time and keeps getting AccessDenied even though IAM looks correct. Which two account/region-level conditions are commonly required before on-demand InvokeModel will succeed? (Select TWO.)
  1. AModel access for that specific model must be granted/enabled in the account for the region being used
  2. BThe model must be available (offered) in the AWS region the request targets
  3. CA Provisioned Throughput commitment must be purchased before any on-demand call
  4. DA SageMaker domain must exist in the account to host Bedrock models
  5. EThe model must first be copied into the account via CreateModelInvocationJob
Show answer & explanation
Correct answer: A, B
On-demand invocation requires that model access is enabled for that model in the account and that the model is actually offered in the target region, both of which are common causes of access errors. Provisioned Throughput is optional, batch jobs do not provision models into an account, and Bedrock does not depend on a SageMaker domain.

Question 2 Amazon Bedrock Fundamentals
A team observes intermittent ThrottlingException errors during traffic bursts on on-demand InvokeModel calls, even though average utilization is well below their quota. They want the simplest change that improves success rates without buying capacity. What should they do FIRST?
  1. ASwitch immediately to Provisioned Throughput for all traffic
  2. BImplement retries with exponential backoff and jitter on throttling responses
  3. CIncrease max tokens so each request does more work and fewer requests are sent
  4. DMove every request to a different region permanently
Show answer & explanation
Correct answer: B
On-demand limits are enforced over short windows, so bursts can throttle even at low average load; retries with exponential backoff and jitter let transient throttles succeed on retry and is the standard first mitigation. Larger max tokens does not reduce request-rate throttling, Provisioned Throughput is a cost commitment that is premature here, and relocating to one fixed region does not inherently add capacity.

Question 4 Amazon Bedrock Fundamentals
An application sends a very large document plus instructions to a Bedrock model and receives a validation error indicating the input is too large. The model itself supports a large context window. What is the MOST likely cause and correct remediation?
  1. AThe request exceeded the model's context window (input plus requested output tokens); reduce input size, summarize/chunk, or lower maxTokens to fit within the limit
  2. BThe error is transient throttling; retry the identical request with backoff
  3. CBedrock caps every request body at 1 KB; compress the JSON to fit
  4. DStreaming is required for large inputs; resend with InvokeModelWithResponseStream unchanged
Show answer & explanation
Correct answer: A
Total tokens (prompt input plus the maxTokens you reserve for output) must fit within the model's context window; exceeding it produces a validation error that you fix by trimming input, chunking, or lowering maxTokens. There is no 1 KB body cap, the error is not throttling, and switching to streaming does not increase the context window.
